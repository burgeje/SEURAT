Tactic Explorer

The tactic library contains a set of tactics that may be employed in a design. All databases generated from SEURAT 6.0.1 will contain a set of tactics selected from “Software Architecture in Practice” (Len Bass; Paul Clements; Rick Kazman). The “Tactic Explorer”(henceforth “explorer”) provides an interface to modify the preset tactics.

The explorer is organized in a tree structure. The visible root is a single node called “Tactic Library”. The children of the root are the positive quality attributes for the tactic in which it improves on (henceforth “categories”). Each tactic may contain additional negative quality attributes, and each tactic may be positively or negatively influenced by the patterns. Those attributes will be shown as children of the tactics.

To add a new tactic, the user right clicks on the root  (or alternatively, the category of which the new tactic improves on) and selects “Add Tactic” in the context menu. A tactic editor will be displayed. The tactic editor should be straightforward to use. One important constraint on the categories is that they must be a leaf of the argument ontology. This is to make sure the tactics will only improve on one quality attribute.

To edit a tactic, the user double clicks on a tactic in the explorer.

You may delete a tactic by right clicking on a tactic and click “Delete”. Note that all children of the tactic will also be deleted from the database.

The impacts of the patterns and negative quality attributes can be added by right clicking on a tactic.

 

 

 

 

 

Tactic Editor:

Tactic Editor is straightforward. Simply enter the name, description, and behavior. If you have right clicked on a category when adding a new tactic or editing an existing tactic, the category is already entered in the editor. Otherwise, you need to select a category before you can save it.

 

Pattern Impact Editor:

The overall score indicates whether the pattern will positively or negatively impact the implementation of the tactic. A score between 0-3 means the tactic will be helped by the pattern. A score of 4 means the tactic will neither be helped nor hindered by the implementation of the pattern (and thus giving a “+1” on the overall rationale of the tactic-alternative under default importance if it is generated). Otherwise, the tactic is being hindered by the pattern.

The auto-scoring feature will only accept valid inputs. (Please refer to the journal article “How do architecture patterns and tactics interact? A model and annotation” (Harrison, Avergiou) if you wish to know the range of valid inputs.)
